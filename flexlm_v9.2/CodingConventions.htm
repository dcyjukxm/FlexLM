1. Introduction

This document discusses the coding conventions that have been adopted by the FLEXlm team. Anyone changing or adding code to FLEXlm should follow these conventions. This only covers C, since C++ is not used in FLEX.

 

As of April 3, 2003 the FLEXlm C developers, Sonny Luu, John Wong and Kirk MacLean, have agreed to follow these coding conventions.

 

This Word doc is checked into the FLEXlm source tree. For convenience I have generated a filtered html page that can be viewed by people that dont have Word. Changes should only be made to the Word document then converted to html.

2. Why do we need these conventions

There are a number of reasons that we need to have coding conventions. A few are listed here:

a.      So someone else can read it

b.     So someone else can understand it.

c.      To make the code easy to follow.

d.     We want to create a base of re-useable. If you write some modules that will be re-used you will probably not wish take the time later to explain it to everyone that will use it.

e.      If we all stick to these coding standards it will be easier for each of us to understand the rest of the code in the project should we need to look at it.

f.        Your code is going to be used for longer than you think, and by more people than you think. Just take a look at how long the [snip] code has been around.

g.     Customers will see some of our source code in the example programs. We need to put forth a well-organized and readable body of example code so our customers dont think we are a bunch of hacks.

3. File header

The following information needs to be placed at the top of the file in a comment block. This should be in every source, include and makefile.

 

This comment block is very important, especially the copyright string. If you edit a file that does not have this comment block then add it. The copyright string needs to be there for legal reasons. $Id: CodingConventions.htm,v 1.3 2003/05/22 22:34:33 kmaclean Exp $ is expanded by CVS to be the name of the file, version and date.

a.      Use #include <> for system includes

b.     Use #include for project includes

c.      DO NOT use relative path names that include ../. Ever.

i. Example, Bad:

5. Comments

5.1. General

Comments are important. If we all wrote code for ourselves we would surely remember all of it and never need to comment any of it. The reality is that we all use modules written by others in the group. We all add enhancements to other peoples code. And as rare as this may be, we have all fixed a bug in someone elses code. Or even more rarely, someone has fixed a bug in your code.

 

a.      If you fix a bug then put your name, date and the GT case or bug number in the comments where the fix was made

b.     The C++ // comments are cool. Dont use them. Some of our code must compile on older systems that do not support this.

c.      Comment each structure and data type that you define with a short comment telling what the struct is used for. Additionally comment each member of the type or struct.

 

/** Send connection and other options to the server */

struct _options {

unsigned long used; /** This is the mask of options that are being changed */

unsigned long value; /** This is the mask of True or False values for the */

/* options that are being set */

} Options ;

 

 

d.     Comment each #define and const. Include what this thing is for and what the value means if it is some magic number. Remember that someone else may not understand the purpose for the #define or const based on the name you chose for it.

 

/** Error exit codes - These are used by the master to tell

* what went wrong with the vendor daemon

*/

#define EXIT_BADCONFIG (BASE_EXIT + 1) /** Bad configuration data */

#define EXIT_WRONGHOST (BASE_EXIT + 2) /** Invalid host */

#define EXIT_PORT_IN_USE (BASE_EXIT + 3) /** Internet "ADDRESS ALREADY IN USE" */

#define EXIT_NOFEATURES (BASE_EXIT + 4) /** No features to serve */

#define EXIT_COMM (BASE_EXIT + 5) /** Communications error */

#define EXIT_DESC (BASE_EXIT + 6) /** Not enough descriptors to */

 

e.      File static and global variables need comments. If the variable is a global (not that there should be any) tell us why.

f.        If your religious beliefs prevent you from putting numbers in #defines or consts and instead putting them directly in the code, then the least you could do is comment your magic numbers.

g.     Write a brief comment for every "logical block" of code. This should be a general, "what & why", comment in plain English, not a technical explanation that merely restates the actual code. Who made the change, what, where, why, how.

h.     As you wind your way through the source and figure out how something works comment it. Put your name and date in there so we can ask you about it later.

 

/***********************************************************

* kmaclean 2/21/03

* The widget gets loaded with the sky hook here then elbow grease is

* applied.

***********************************************************/

 

i.        Write a comment where you do something TRICKY, or wherever you make some system call that isn't self-explanatory, or that's required for some reason that someone else might not know. Protect yourself from someone (maybe yourself) changing this code later because it "looks" wrong.

 

5.1.1. Tricky example comment

void ShutDownStuff()

{

Stuff*entity;

 

entity = GetFirstEntity();

while (entity)

{

RemoveEntity(entity);

 

/* Something tricky here?

If we removed the first entity then the next entity is now the first entity so get the new first entity and remove it */

entity = GetFirstEntity();

}

If the comment was not here one may be inclined to say that this is a bug, because we cant get the first entity after deleting it, and fix it by changing the code to GetNextEntity() thus causing three people to spend hours wondering why the Stuff manager is now crashing.

5.2. Comment each procedure

Place a comment block at the beginning of each procedure. Even if you think it is obvious.

a.      Include a sentence or two describing what the procedure does.

b.     All the parameters should be listed along with a description of how they are used.

c.      The return value is and what it means.

d.     This includes procedures that are just temporary hacks. Comment it as a hack, why it is a hack, and when it will be going away. Also indicate what the right thing to do is.

e.      Place a line of stars or dashes or whatever at the beginning of each procedure. Many of us have joined the twentieth century and use editors that color code the source. This makes it easy to scan the file and find the beginning of a procedure.

 

If you dont spend the two or three minutes putting in the comment block then each time someone wants to use your code you can spend five minutes for each routine in the file explaining it to each person.

7. Naming

a.      Use meaningful names for variables and class members. It helps when looking at the code to see something like stringInfo.size = 3 as opposed to s.s = 3;

b.     Avoid using unqualified instances of size and length when referring to character strings because it is ambiguous whether these are byte is char sizes.

c.      Use all caps for #defines, typedefs, consts and enums to distinguish them easily. If used properly enums are ok for internal use. Its really better if they dont cross the API boundary since the size is not defined..

7.1. Procedures and variables

a.      We mix upper and lower case by capitalizing "word boundaries" within the name (i.e. lc_NextConf),

b.     Static functions start with s_.

c.      Internal functions start with l_.

d.     External API functions in the client library start with lc_

e.      Use meaningful names so the reader can understand what this variable is for.

 

Bad

Good

Void cp( XYZ *a, XYZ *b)

 

Void s_copyXYZ( XYZ *destxyz,

XYZ *srcxyz)

 

7.2. Prefixes

Placing a prefix on a name helps ensure that there are no name conflicts with other portions of the code. It also lets others know that these things are related. We

Use prefixes instead of suffixes so that common symbols and files sort together.

a.      When naming files if you have a number of files that are related name them all with the same prefix or put them in a separate directory.

b.     Put a prefix on groups of #defines and consts that are related. Prefix all the elements in an enum the same.

8. Type usage

a.      The type used depends on what you are writing for.

b.     Types should be appropriate for interface or project working on.

c.      Non-windows should use standard c-types or typedefed types.

i. Don't use Windows types unless you're calling a Windows routine.

8.1. Boolean

a.      If you mean True or False then use the M_BOOL type.

b.     If you want to return success or failure from a procedure then use M _BOOL not int. This makes it easier for anyone reading your code to know that you intended to return success or failure.

c.      If you're using a Boolean variable, just use "if (bool)" and "if ( ! bool)" as tests. Don't test for equality against TRUE or FALSE. A boolean may be set to 1 or 0xFFFFFFFF to indicate True.

8.2. Macrovision abstracted types

Use Macrovision abstracted data types instead of primitives. For now we only have the types below. Later we will add more abstracted types.

 

typedef unsigned short M _CHAR_UNICODE;

typedef unsigned char M _CHAR_UTF8;

typedef unsigned int M _BOOL;

 

#define M_TRUE ((M_BOOL) 1)

#define M _FALSE ((M _BOOL) 0)

 

 

 

8.3. Enums

If you are going to use enums then use them properly. Enums are ok for internal use. Since the size is undefined it would be better if they didnt cross the API boundary.

a.      Name them, or even better typedef them.

b.     Use the enum type when declaring a variable for the enum. The debugger will display the enum element name if the variable is typed correctly.

 

/* Return value used by routines that read and write to the net */

typedef enum {

NR_CLOSED, // connection is closed

NR_SUCCESS, // The operation was successful

NR_NO_DATA, // The read or write did not read or write all of the data

} NR_VAL ;

 

Bad

Good

int func(void)

{

int val;

 

if (something_is_wrong)

val = NR_CLOSED;

return val;

}

NR_VAL func(void)

{

NR_VAL val;

 

if (something_is_wrong)

val = NR_CLOSED;

return val;

}

9. Formatting

9.1. General

a.      Place external functions at the top of the file and statics at the bottom. Use a comment block to separate this section from the external code

b.     Do not place defines inside a struct declaration. It makes things too hard to read. Instead make a comment for the struct member to reference a set of defines.

c.      Remove dead code. Dont #if 0 or comment it out. It is sometimes useful to leave old code in there for reference. If you feel you need to do this then put in comments to tell us why this code is no longer used.

d.     DO NOT put special characters in source code. This is sometimes done to put in bullet items and other formatting. Dont do it.

 

9.2. Indentation and Spacing

9.2.1. Tabs

a.      Use tabs to indent.

b.     Tab setting is 4.

c.      Do not use spaces.

d.     Do not mix tabs and spaces. For any reason. Under any circumstances.

e.      Use one tab per block. Not one for the brace and one for the code

Bad

Good

If (abc == 5)

{

hellothere(abc);

}

If (abc == 5)

{

hellothere(abc);

}

 

f.        Do not use an editor that replaces tabs with spaces.

g.     Tabs are our friends. Using tabs allows you to set smaller tab stops to squish more on the screen if needed. This is especially useful in the debugger.

 

9.2.2. White space

a.      Use spaces to make your code more readable.

b.     Dont squish all your code together. Use white space to separate logical blocks of code.

c.      One statement per line. Not only does this make it easier to read the code it makes it easer to debug.

d.     Do not put statements on the same line as a conditional. It makes it harder to debug because you cant tell if the condition was true just by stepping.

 

Bad

Good

If (xyz = 24) x=1;

if (stuff == 3) x = 2;

else x=3;

If (xyz = 24)

X = 1;

If (stuff)

X = 2;

Else

X = 3;

 

If (stuff)

{

x=2;

}

If (stuff)

{

x=2;

}

If (stuff)

{

x=2;

}

If (stuff)

{

x=2;

}

9.3. Expressions

a.      If you make an assignment inside of a comparison it can be error prone and confusing. Use an explicit comparison.

Bad

Good

If (xyz = getXyz(abc))

if ((xyz = getXyz(abc)) != 0)

 

 

9.4. Functions

a.      Do not squish code against the left edge within a procedure. Always indent code and variable declarations within a procedure. This makes it easier to find the beginning of a procedure when you are scanning the file.

b.     If a variable must be cleaned up on return from a function then declare it at the beginning to make it easier to clean up on an error return.

c.      Do not duplicate code in different routines. Make small functions to do it.

d.     Initialize variables when you declare them, unless it doesnt make sense to do so, such as when you are declaring a structure or something. In that scenario, make sure to initialize it ASAP. Esp. pointers.

e.      Some of our code pointers to -1. Dont do this. It is bad. Initialize pointers to NULL.

 

9.4.1. Prototypes

a.      Always specify the return type

b.     If the function is not used outside the file then make it static. . This helps the optimizer. And forces the developer to think if they ever want to make a private all public.

c.      Use ANSI function prototypes not K&R. Much of the code does not do this now. As we edit files we need to change this

 

Bad K&R

Good ANSI

MyFunc ( abc, def)

Int abc;

Char * def;

{

}

Static void MyFunc (int abc, char * def)

{

}

 

9.4.2. Length

Some of our code currently has functions that are thousands of lines long.

a.      Keep functions a reasonable length. If you think your function needs to be long then youre doing it wrong. J

b.     Many people have said that the entire function should fit on the screen. This is a little extreme.

9.5. { } usage

a.      Use them on all loops, even if there is only one statement.

Bad

Good

For (index = 3, ptr = string;

Index < 27, *ptr != \0;

*ptr += index)

Index++;

 

For (index = 3, ptr = string;

Index < 27, *ptr != \0;

*ptr += index)

{

Index++;

}

 

b.     Use them on all multi-line conditionals, even if there is only one statement.

Bad

Good

if ( *localString == 0 &&

( *(localString + 1) == 'x' ||

*(localString + 1) == 'X'))

localString += 2;

if ( *localString == 0 &&

( *(localString + 1) == 'x' ||

*(localString + 1) == 'X'))

{

localString += 2;

}

9.6. Macros

a.      Macros should be simple.

b.     Macros make the code harder to debug. The debugger cant step into them.

c.      Macros in some of our code are way over used. Especially in FLEXlm.

d.     In FLEXlm macros have been used to override standard library functions. For example:

#define strcpy(a,b) l_strcpy(a.b)

This is bad. Dont do it.

e.      NO macros greater than 2 lines

f.        If it needs to be longer then it should be a procedure. Make it static inline, if its short, to help the compiler optimize it. Inline is only a suggestion to the compiler so it may not do anything anyway. I thought it was part of the language though?

g.     Comment the dang things.

h.     Use () around parameters in macros. Only the last #define in the example gives the expected result.

 

Definition

Results using: done = DOIT(xyz + abc) * 5;

#define DOIT(a) 57 + a

Done = 57 + xyz + (abc * 5);

#define DOIT(a) 57 + (a)

Done = 57 + ((xyz +abc) * 5)

#define DOIT(a) (57 + (a))

Done = (57 + (xyz + abc)) * 5

 

i.        Do not use macros as a shortcut for typing out the entire variable or struct member name. When youre in the debugger you cannot get at the struct member name through the macro.

 

#define D(_x) (data+ (_x*(MAX_CONFIG_LINE+1)))

 

func()

{

char *p1, *p2;

p1 = D(FNAME); p2 = D(DAEMON);

}

11. Globals

a.      Dont use globals, ever.

b.     If you need global data then make the variable file static and use C accessor functions.

c.      If for some bizarre reason you felt it necessary to put a global in. Be prepared to provide notarized depositions from five witnesses, signed in blood, verifying why the variable absolutely must be global.

 

12. Other Miscellaneous Rules

a.      Avoid using an extern in a source file to gain access to a function from another module. Instead, put the extern declaration in the proper header, this reduces work later on if the variable or function has been referenced from several other modules but needs to be changed

b.     Do not place magic numbers in your code. Use a #define or const for them. Comment them. If you spent two days tweaking to come up with some number then it deserves a large comment describing what the number is for, how you came up with it, and why no one should change it. Or better yet let it be calculated by the compiler, as much as is possible.

c.      Single point of exit for all functions if reasonable.

d.     Put .h files in the h directory. Not the local directory.

e.      Modularize memory allocation and de-allocation. Create allocation and free routines for your structs. This way malloc() and free() will proliferate through out the code.

f.        Use our memory allocation routines l_malloc() and l_free().

g.     No @$*#ing #ifdefs all through the code. Put #ifdefs in the header if you must. Create abstracted routines and compile in different code depending on the platform. If you think this isnt necessary then go try to understand src/lm_getid_typ.c.

h.     sprintf and scanf are nice and easy for formatting and parsing. They are not safe and they are slow. Dont add more.

i.        Use #ifdefs in include files to prevent the code from being included multiple times.

 

#ifndef INCLUDE_MYFILE_H

#define INCLUDE_MYFILE_H

include file stuff here

#endif

j.        Strcmp() returns zero if the strings are equal. Ive seen many places in the code like this:

if ( ! strcmp( a, b))

This is bad. On casual reading of the code the ! implies that on failure the code in the if gets executed. Explicitly comparing to zero is much clearer

if ( strcmp(a, b) == 0)

Better yet use the L_STREQ() or L_STREQ_N() macros.

13. Testing

Ideally we should compile and test our code before checking it in. This is a pain because its difficult to move your source from, say Windows to LINUX. How to resolve this?

a.      Immediately upon check in compile and test on other platforms. At least Windows, Solaris and LINUX.

b.     If you discover that something is broken then send e-mail to others on the team and tell them.

14. Source Control Rules

The following rules all apply to using CVS and will hopefully result in as little friction as possible on such a large project:

a.      When checking in your changes, first do a cvs update then compile and test to make sure your code works with what is checked in and to make sure you do not overwrite someone elses changes.

b.     Always, always, always compile and test something before you check it in.

c.      When CVS asks for comments, enter comments that describe what you did. A comment of Fixed some stuff, is not sufficient. Also put GT bug/case number in comment.

d.     If you check in one file, check in ALL your files. Otherwise, you'll probably introduce errors since other programmers will only have part of your changes.

e.      When you add a new file to the project, please try to be extra careful in making sure that you've checked in ALL the relevant files: include files, data files, etc.

f.        If you check in a number of files then make a new directory and do a clean cvs get and a build.

g.     If you check something in and it causes the project not to compile or link or run. Send mail stating this and indicate when it should be fixed. Send mail when it is fixed.

h.     Compiler warnings must be weeded out of a module before it is checked in. This ensures that when something pops up during compile time, there is something wrong that definitely needs to be looked at (if theres seventy warnings in a build because programmers get sloppy on this, its hard to pick out one or two potentially important warnings as the build progresses). Id suggest Visual C warning level 3 or equivalent.

15. Guidelines to creating re-useable code

a.      Isolate your code from the project. Put it in a separate file or directory and create a subproject in CVS so that it will be easy for someone else to grab.

b.     Make your API simple and consistent.

c.      Consider requiring the user to provide callback functions if your module needs to access functions in the rest of the project.

d.     Do not use any project globals.

e.      Ideally there should be no dependencies on the rest of the project. In practice this is not usually possible. So just try to minimize them. Do not include every .H file in the project.

f.        If there are dependencies then comment them in the file header.

g.     Use lots of comments.

16. How to update the current code

As you edit the current source files do the following:

a.      Update all the function prototypes and declarations to ansii style.

b.     If you figure out what a function or code block does comment it.

c.      Where there are multiple statements on a single fix it.

d.     Indent comments to the same level as the block they apply to.

e.      NO MORE #ifdefs

f.        On windows set the warning level to 3 temporarily and weed out as many warnings as you can.

g.     We should start hiding data. Anything new that is added to a struct that is exposed to the client but is not for their use should be added as an opaque struct pointer.

h.     If you find an include that does not have #ifdefs around it then add them

#ifndef INCLUDE_MYFILE_H

#define INCLUDE_MYFILE_H

include file stuff here

#endif

17. The Future

Things that want to do but are in the future.

a.      We need to define a file naming convention.

b.     It would also be a good idea to document all the current defines used in the code. For example #ifdef PC and #ifdef WINNT are sometimes synonymous and sometimes not. Any volunteers?

c.      It would be nice to add debug trace statements at entry and return from each function, which would provide info such as function name and parameter values, file, line number. This would be a very valuable debugging tool. Of course, it would be a major pain to have to add this to existing code. Entry points are easy but return points will be more difficult

 

This could even be expanded to keep a running stack of where you are. This stack can be dumped on a memory fault or saved to the log on an error.

d.     Extend our data hiding to include the old structs like LM_HANDLE.
